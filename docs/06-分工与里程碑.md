## 06 · 分工与里程碑

### 1. 分工原则
两人配合，按"服务端 + 对接 owner"和"前端 + UE/交互 owner"划分，职责有重叠、互不等于完全隔离。每个模块必须两人一起联调一次（15 分钟内能跑通即可）。

> 下面用 **A**、**B** 指代两位开发。具体谁是 A 谁是 B 由你们自己决定。
> A = 后端主理（Cloud 域），B = 前端主理（Miniprogram 域）。

### 2. 两人的职责边界

**开发者 A（后端 + 管理后台 主理）**
- `backend/` FastAPI 工程：用户/资料/择偶/推荐/收藏/联系/通知/管理 路由模块
- `models.py` 数据库 schema（SQLModel）、Alembic migrations
- Redis 缓存策略（每日推荐结果、限流、计数）
- 定时任务（APScheduler 嵌入 / 后期升级 Celery）
- 接口契约维护（FastAPI 自带 OpenAPI 文档 → 04/09 文档配合）
- 微信开放接口对接：`jscode2session`、订阅消息、内容安全
- `admin/` React 后台的业务页面（资料审核 / 解锁记录 / 用户管理 / 数据看板）—— 复刻模板已有 `items.tsx` 模式即可，工作量小
- 部署：模板 docker-compose + Traefik + GitHub Actions
- 域名 + ICP 备案 跟进（一期最关键的非技术任务）

**开发者 B（前端主理）**
- `miniprogram/pages/*` 页面逻辑、状态管理
- `miniprogram/services/*` 服务层封装（调云函数）
- `miniprogram/utils/*`、`miniprogram/constants/*` 公共逻辑
- 组件抽取（如必要把 home/detail/profile-edit 拆成 `components/screen-*`）
- 微信 API：订阅消息、分享、图片上传、剪贴板、页面路由
- UI/UE 细节、样式修正、资源管理（图标、占位图）
- `app.js`、`app.json`、`project.config.json` 的小程序侧配置
- 错误提示、loading、空态 UI

**两人共同负责**
- 联调、bug 跟踪
- [docs/](./) 文档更新，任何人改接口/字段 → 立刻改文档
- [docs/08-待确认事项.md](08-待确认事项.md) 的关闭
- Code Review：每个 PR 另一方 review 通过后才能合并

### 4. 日常节奏
- 每日 10min 同步：昨天做了啥、今天做啥、有啥阻塞（站会）
- 每 M 结束做一次 15min demo
- 接口契约冲突必须 M0 之前谈清；M1 后修改接口要同步到 04 文档 + 双方

### 5. 任务卡粒度建议
用任意轻量工具（微信群 TODO / Notion / GitHub Issue）拆任务，每张卡以"可独立合并、能写单独 commit"为粒度。命名建议：
- 后端：`[cf][profile] 实现 saveProfile 接口`
- 前端：`[mp][detail] 对接 match.getProfile`
- 联调：`[link][M2] detail 联调全量字段`

### 6. 风险清单
| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| 接口契约后改 | 前后端返工 | 04 + 09 文档是 single source of truth，改接口同步改文档；NestJS Swagger 自动出 OpenAPI 文件可校对 |
| 域名 ICP 备案周期 | M0 直接堵 7-20 天 | 项目启动**第一天**就提交备案，备案期间用云托管/云函数兜个临时域名做开发 |
| 服务器被恶意请求 | 数据泄露/账单暴涨 | 接口加限流（@nestjs/throttler）、Redis 计数；Nginx 设连接数；ufw 只放必需端口 |
| 数据库备份缺失 | 数据丢失风险 | 每天 cron `pg_dump` 推到 COS，保留 30 天 |
| 订阅消息模板审核周期 | M4 堵塞 | M0 阶段就把模板申请提交 |
| 资料图片内容安全 | 上线被拒 | 提交前用 `msgSecCheck` 灰度一遍；desc 先调一次 |
| 寻缘号冲突 | 注册失败 | 8 位数字空间 1 亿，冲突概率极低；重试 3 次即可 |
| 定时任务在开发环境不跑 | M4 验不了 | 先手动调用 `notify.buildDailyRecommendations` 验证逻辑，再挂定时器 |
