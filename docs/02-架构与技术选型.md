## 02 · 架构与技术选型

> 📌 本文档原方案为微信云开发。**最终方案是基于 [full-stack-fastapi-template](https://github.com/fastapi/full-stack-fastapi-template) 二次开发**（FastAPI 后端 + React 管理后台一体化），详见 [09-后端框架选型.md](09-后端框架选型.md)。本文档保留作为参考，下方"云函数"小节请理解为"FastAPI 路由模块"。

### 1. 总体架构
```
┌──────────────┐     wx.cloud.callFunction     ┌───────────────────┐
│  小程序端     │  ───────────────────────────▶ │ 云函数 (Node.js)   │
│  (miniprogram)│                                │  按业务域拆分      │
│              │  ◀─────────────────────────── │                   │
└──────────────┘         JSON 结果              └────────┬──────────┘
       │                                                │
       │ wx.cloud.uploadFile / downloadFile             │ 读写
       ▼                                                ▼
┌──────────────┐                              ┌───────────────────┐
│  云存储       │                              │  云数据库(文档型)   │
│  (头像/图片)  │                              │  user / profile…   │
└──────────────┘                              └───────────────────┘
```
- 前后端之间的唯一通道：**云函数调用 + 云存储**。小程序端不直接读写数据库（除极少数只读公开集合），一切走云函数鉴权。
- `wxContext.OPENID` 作为用户唯一标识。`寻缘号` 单独生成（见 03 文档）。

### 2. 技术栈
| 层 | 选型 | 说明 |
| --- | --- | --- |
| 小程序 | 原生 WXML / WXSS / JS | 已经选用，继续保持 |
| 云函数 | Node.js 16+ / `wx-server-sdk` ~2.4 | 与当前模板一致 |
| 数据库 | 微信云开发 JSON 文档库 | 开发快，权限好做 |
| 文件 | 微信云存储 | 与数据库同域，`fileID` 直接在组件里用 |
| 订阅消息 | 微信订阅消息 | `wx.requestSubscribeMessage` |

### 3. 目录与命名（最终：分仓 2 个文件夹）

详细结构与说明见 [09-后端框架选型.md](09-后端框架选型.md) 第 6 节。

```
/data/gzl/project/
├── qianchenhunlian/                  仓 1 · 小程序
│   ├── miniprogram/
│   │   ├── pages/
│   │   ├── services/        [新增] 对后端 HTTP API 的封装
│   │   ├── utils/           [新增] request.js / date.js / format.js
│   │   └── constants/       [新增] 枚举（学历/收入/婚况），与后端保持一致
│   └── docs/                所有规划文档
│
└── qianchenhunlian_server/           仓 2 · 服务端整体
    ├── backend/             FastAPI（来自模板，按业务域划分 routes）
    │   └── app/api/routes/
    │       ├── wechat.py    小程序登录、订阅消息上报
    │       ├── profiles.py  我的资料 + 择偶要求
    │       ├── matches.py   推荐 / 筛选 / 详情
    │       ├── favorites.py 收藏 / 看过我
    │       ├── contacts.py  解锁 / 发意向
    │       ├── admin/       审核 / 权益发放
    │       ├── login.py     管理员邮箱密码登录（模板自带）
    │       ├── users.py     用户管理（模板自带）
    │       └── utils.py
    ├── admin/               React 后台（来自模板 frontend/）
    ├── data/                持久化：postgres / redis / uploads
    ├── docker-compose.yml
    └── scripts/
```

**约定**：
- 每个云函数一个目录，`index.js` 里用 `event.action` 路由到子处理器，保持接口集中，避免云函数数量爆炸。示例见 04 文档。
- 所有云函数统一返回 `{ code, data, msg }` 结构（见 04 文档错误码段）。
- 前端 `services/*.js` 封装调用，页面只 import service 函数，不直接写 `wx.cloud.callFunction`。

### 4. 页面结构是否拆分
当前所有屏幕挤在一个 `pages/index`，好处是切换快、状态共享方便；坏处是单文件 700 行，后续上 tabbar / 支付 / 聊天时会膨胀。

**建议（一期不强求）**：保留单页结构，但把每个屏幕的 wxml 抽成组件 `components/screen-*`，让 `index.wxml` 只做屏幕调度。如果开发过程中页面复杂度真的顶不住，再拆 pages。

### 5. 环境
- **开发环境**：每位开发在微信开发者工具里开自己的云开发环境（免费配额够用），互不干扰。
- **测试/生产**：正式环境由 owner 申请，填入 `app.js` 的 `globalData.env`，通过工程分支切换。
- `envList.js` 保留当前结构，后续可用于多环境切换。

### 6. 安全与权限
- 所有写操作必须经云函数，读取他人资料也走云函数（要脱敏 + 解锁校验）。
- 数据库集合权限一律设为"仅创建者可读写"，然后通过云函数用管理员身份访问。
- 联系方式（手机号/微信号）字段永远不下发到未解锁用户，详见 04 的 `match.getProfile` 返回字段差异。
