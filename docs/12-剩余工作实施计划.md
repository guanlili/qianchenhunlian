## 1. 项目约定（执行前必读）

### 1.1 技术栈与目录

- 后端：FastAPI + SQLModel + PostgreSQL，代码在 `backend/app/`；管理端路由集中在
  `backend/app/api/routes/admin.py`，小程序端路由在同目录其它文件。
- 前端（后台）：React 19 + Vite + TanStack Router/Query + shadcn/ui，代码在 `admin/src/`；
  文件式路由在 `admin/src/routes/_layout/`。
- 小程序：原生微信小程序，`miniprogram/`；接口调用统一走 `miniprogram/services/` 封装层。

### 1.2 硬性守则（违反即打回）

1. **禁止改动** `admin/src/routes/index.tsx`（对外官网落地页，与后台无关）。
2. **禁止手改** `admin/src/client/*`（自动生成）与 `admin/src/routeTree.gen.ts`。
   改了后端接口后按 1.4 重新生成。
3. 数据库加列**必须**同时登记到 `backend/app/core/db.py` 的 `_PENDING_ADD_COLUMNS`
   （本项目建表走 `init_db` 的 `create_all`，老库加列靠这个列表，不走 alembic）。
4. 敏感写操作（工单处理/改派、反馈处理、批量赠送、配置修改）必须调用 `admin.py` 里的
   `_audit(session, actor, action, target_user_id, detail)` 记审计日志，与业务变更同事务提交。
5. 权限模型：`require_admin`（仅 superuser）/ `require_admin_or_staff`（含员工），
   **定义在 `backend/app/api/deps.py`**，新端点照抄 `admin.py` 内已有端点的依赖写法与 import；
   红娘（matchmaker）数据范围用 `_store_owner_id(actor)` + `_check_store_scope`，
   总部员工（hq_staff）不可见联系方式、不可代录（沿用 `_can_see_contact` / `_check_edit_permission`）。
6. 新端点的 response_model 一律定义 SQLModel DTO（参考 `AdminMemberItem` 写法），
   分页参数统一 `skip` / `limit`，列表返回 `{items, total}`。
7. 前端新页面遵循现有模式：`useQuery` + `AdminService.xxx()`；写操作 `useMutation` 成功后
   `invalidateQueries`（相关 queryKey 见各任务）；组件用 `admin/src/components/ui/` 里的 shadcn 组件。
8. 提交信息格式沿用仓库惯例：`feat(backend): ...` / `feat(admin): ...` / `feat(mp): ...`。

## 2. P2 任务（优先做，按顺序）

### 任务 A：撮合工单升级（后端）

**改动文件**：`backend/app/models.py`、`backend/app/core/db.py`、`backend/app/api/routes/admin.py`、`backend/app/api/routes/contacts.py`

1. `ContactRequest` 表加字段 `store_id: uuid.UUID | None = Field(default=None, index=True)`
   （归属门店）。同时在 `db.py` 的 `_PENDING_ADD_COLUMNS` 登记 `("contactrequest", "store_id", "UUID")`。
2. 小程序端创建工单时（`contacts.py` 的 `submit_contact_request`）：
   `store_id` 默认取发起方 profile 的 `home_store_id`（可能为 None）。
3. `admin.py` 修改：
   - `GET /admin/contact-requests` 增加过滤参数 `store_id: uuid | None`；
     红娘（`_store_owner_id` 非 None）只能看 `store_id == 本店` 的工单。
     响应项 `AdminContactRequestItem` 增加 `store_id`、`handled_by`、`overdue: bool`
     （`status=='pending'` 且 `created_at` 距今 > 48h）。
   - 联系方式字段：非 superuser 且非本单归属店红娘 → 置 None（复用现有脱敏思路）。
   - `POST /admin/contact-requests/{id}/handle` 权限从 `require_admin` 放宽为
     `require_admin_or_staff`；红娘只能处理本店工单（否则 403）；记审计
     `_audit(..., "handle_ticket", target_user_id=req.from_user_id, detail={"request_id","status","note"})`。
   - 新增 `POST /admin/contact-requests/{id}/assign`（仅 `require_admin`）：
     body `{store_id: uuid | None}`，校验门店存在且 active，记审计 `assign_ticket`。
4. 重新生成前端 client（见 1.4）。

**验收**：superuser 能改派工单到某门店；红娘登录只见本店工单且能处理；hq_staff 能处理但看不到双方联系方式；pending 超 48h 的单返回 `overdue=true`；以上操作均出现在 `GET /admin/audit-logs`。

### 任务 B：撮合工单升级（前端）

**改动文件**：`admin/src/routes/_layout/requests.tsx`、`admin/src/components/Requests/`

1. 列表按状态分 Tab：待处理 / 已同意 / 已拒绝 / 已建群 / 已关闭（参考 `review.tsx` 的 Tabs + 角标写法，
   待处理数取 `adminStats().pending_tickets`）。
2. 每单卡片：双方摘要（寻缘号/性别/年龄/居住地，点击跳 `/members/$userId` 360° 详情）、
   留言、归属门店（门店名映射参考 `members.tsx` 的 `storeMap`）、`overdue` 为 true 时整卡红色边框 + "已超时"徽标。
3. 处理对话框（现有 `HandleDialog` 扩展）：状态流转按钮 + 备注；处理成功后
   invalidate `["admin-requests"]`、`["admin-stats"]`。
4. superuser 可见"改派门店"按钮 → 弹窗选门店（参考 `components/Members/AssignStoreDialog.tsx` 直接改造复用）。
5. 时间线展示：卡片展开显示 创建时间 → handled_at + admin_note + 处理人。

**验收**：`npm run build` 通过；页面上完成一次 pending → accepted → contacted 的完整流转并能看到时间线。

### 任务 C：意见反馈处理流

**改动文件**：`backend/app/models.py`、`backend/app/core/db.py`、`backend/app/api/routes/admin.py`、`admin/src/routes/_layout/feedback.tsx`

1. `Feedback` 表加 `admin_note: str | None (max 255)`、`handled_by: uuid | None`、`handled_at: datetime | None`；
   `_PENDING_ADD_COLUMNS` 登记三列（`VARCHAR(255)` / `UUID` / `TIMESTAMP`）。
   注：`TIMESTAMP`（不带时区）是正确写法——本项目全库时间列统一 naive UTC
   （已实测 `timestamp without time zone`），且列表里已有先例 `("profile", "verified_at", "TIMESTAMP")`，照抄即可。
2. 新端点 `PUT /admin/feedback/{id}`（`require_admin_or_staff`）：
   body `{status: Literal["open","handled"], admin_note: str | None}`；写 `handled_by/handled_at`；
   记审计 `handle_feedback`。`GET /admin/feedback` 响应项补这三个字段 + 状态过滤参数。
3. 前端 feedback 页：状态筛选（全部/未处理/已处理）、行操作"标记处理"弹窗（填备注）、
   已处理行显示备注与处理时间；用户列点击跳 `/members/$userId`。
   成功后 invalidate `["admin-feedback"]`、`["admin-stats"]`（工作台的未处理反馈角标依赖它）。

**验收**：处理一条反馈后工作台"未处理反馈"数字减 1，审计日志有记录。

### 任务 D：门店管理增强

**改动文件**：`backend/app/api/routes/admin.py`、`admin/src/routes/_layout/stores.tsx`（+ 可新增 `stores.$storeId.tsx`）

1. 新端点 `GET /admin/stores/{store_id}/overview`（`require_admin_or_staff`，红娘限本店）：
   返回 `{store: StorePublic, member_count, verified_count, pending_tickets, matchmakers: [StaffPublic]}`
   （member_count = `Profile.home_store_id == store_id` 计数；matchmakers = `Staff.role=='matchmaker' and store_id` 匹配）。
2. 前端门店详情页 `/stores/$storeId`：门店信息 + 三个统计卡 + 红娘列表；
   superuser 可在此页直接创建红娘账号（复用 `components/Staff/AddStaff.tsx`，
   role 固定 `matchmaker`、store_id 预填当前门店——AddStaff 若无 role/store 字段则补上，
   同时员工管理页的创建表单加 role 下拉：`hq_staff`/`matchmaker`，选 matchmaker 时必选门店）。
3. 门店列表行点击进入详情页。

**验收**：从门店详情页建一个红娘账号，用它登录，只能看到本店会员/工单。

## 3. P3 任务

### 任务 E：权益管理页（全局流水 + 批量赠送）

1. 后端：`GET /admin/transactions`（`require_admin`）：全局 `BalanceTransaction` 分页，
   过滤 `source` / `user_id`（可传寻缘号需先查 user）/ 时间范围 `from_date` `to_date`（YYYY-MM-DD）；
   响应项带 `xy_code`（join User）。
   `POST /admin/grants/batch`（`require_admin`）：body
   `{delta: int, reason: str, filter: {verified: str | None, audit_status: str | None, store_id: uuid | None}}`，
   圈中用户逐个加余额 + 写流水（`source="admin_grant"`, note=reason）+ 一条审计
   `batch_grant`（detail 记 filter、delta、命中人数）。上限保护：命中 > 500 人时返回 400。
2. 前端新页 `/benefits`（导航"权益管理"，仅 superuser 可见——参考 `AppSidebar.tsx` 的 `adminItems`）：
   流水表（来源用中文标签，映射表抄 `members.$userId.tsx` 的 `TXN_SOURCE_LABEL`）+
   "批量赠送"按钮 → 弹窗（条件 + 数量 + 原因 + 二次确认，显示预估命中人数——
   可先调 `listMembers` 相同条件拿 total 展示）。

**验收**：按"已实名"条件批量赠 2 次，相关会员余额与流水正确，审计日志有 `batch_grant`。

### 任务 F：数据统计页

1. 后端：`GET /admin/stats/daily?from_date=&to_date=`（`require_admin_or_staff`）：
   按日聚合（默认最近 30 天）：`signups`（User.created_at）、`profiles_submitted`（Profile.created_at）、
   `audits_passed`（AuditLog action=audit_pass 按日）、`tickets_created`（ContactRequest.created_at）、
   `tickets_succeeded`（status in accepted/contacted 且 handled_at 当日）。
   返回 `{days: [{date, signups, profiles_submitted, audits_passed, tickets_created, tickets_succeeded}]}`。
   红娘维度不做（一期全局）。
2. 前端新页 `/analytics`（导航"数据统计"，全员可见）：
   - 趋势：不引第三方图表库，用现有 CSS/div 柱状（高度按比例）或简单 SVG 实现即可；
   - 漏斗卡片：注册 → 有资料 → 审核通过 → 发起过工单 → 撮合成功（一次性聚合数，
     后端在同端点多返回一个 `funnel` 对象）；
   - 门店对比表：复用任务 D 的 overview 数据或新增轻量聚合。

**验收**：页面能展示最近 30 天趋势与漏斗，数字与数据库一致（抽查 2 项）。

### 任务 G：平台设置（公告 / 协议文案）

1. 后端：通用端点 `GET /admin/site-settings/{key}`、`PUT /admin/site-settings/{key}`

3. 小程序：`pages/agreement`、`pages/privacy`、`pages/store-protocol`、`pages/refund-protocol`
   改为启动时从公开端点拉取文案，拉不到时回退现有本地文案（不能白屏）。

**验收**：后台改用户协议文案 → 小程序对应页面刷新后生效；后台记审计。

## 5. 交付与审核流程

1. 按 任务 A → K 顺序执行，**一个任务一个 commit**，格式如 `feat(backend): contact request store assignment (task A)`。
2. 每个任务完成后自查：1.3 的构建/检查全绿 + 该任务"验收"条目逐条过。
3. 不确定的设计决策：**不要自行发挥**，在 commit message 或单独 NOTES.md 里列出问题留给审核。
4. 全部完成后交审核（审核人会跑 P1 的冒烟测试脚本思路 + 逐任务验收 + code review）。
