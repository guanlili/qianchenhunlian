## 一、产品概述

### 1.1 产品定位

乾缘婚恋是一个**线上线下结合的严肃婚恋服务平台**：

- **线上**：微信小程序承载会员注册、资料展示、每日推荐、互相心动、申请联系等轻量交互；
- **线下**：实体门店（红娘团队）承载实名核验、人工撮合、一对一服务，建立信任壁垒。

区别于陌生人社交产品的核心差异：**不做自由聊天，所有联系建立均经过平台（红娘）审核撮合**，保证会员质量与安全，突出"靠谱、真实、有人服务"的心智。

### 1.2 目标用户

- **C 端会员**：有明确婚恋诉求的单身人士，以及代子女相亲的父母（产品内含"为孩子找""为亲友找"关系选项）；首期地域聚焦山东省内各地市。
- **B 端使用者**：总部运营/审核人员、各门店红娘。

### 1.3 业务模式

| 阶段 | 模式 | 状态 |
| --- | --- | --- |
| 当前 | 免费注册 + **解锁次数**权益（注册赠送，后台可赠送/充值），申请联系消耗次数 | ✅ |
| 本期 | 维持解锁次数模式，完善权益流水与后台管理 | 🔨 |
| 远期 | 微信支付购买次数/会员套餐、线下门店服务收费录入 | ⏳ |

| 角色 | 端 | 说明 |
| --- | --- | --- |
| 游客 | 小程序 | 未完善资料的新用户，可浏览推荐（脱敏），不能发起联系 |

## 三、核心业务流程

### 3.1 会员生命周期

```
微信授权登录（自动注册，分配唯一"寻缘号"）
  → 填写相亲资料 + 择偶要求（可选：父母代填信息）
  → 提交审核（后台人工审核照片与资料）
  → 审核通过，进入每日推荐池
  → （可选）到店实名核验，获得实名标识
  → 浏览 / 收藏 / 心动 / 申请联系
  → 红娘撮合成功 → 线下见面
```

### 3.2 联系建立流程（平台核心流程）

```
会员 A 在详情页点击"申请联系"会员 B
  → 消耗 1 次解锁次数，生成撮合工单
  → 红娘介入：联系 B 确认意愿
      ├─ B 同意 → 红娘建群/互推联系方式 → 工单完结（撮合成功）
      └─ B 拒绝 → 工单关闭（次数不退还，防恶意申请）
  → 全程小程序端可查看工单进度
```

> 设计要点：次数在**提交申请时**即扣除，被拒不退还，与主流平台一致，防止骚扰式申请。

### 3.3 资料审核流程

```
会员提交/修改资料 → 状态"审核中"（推荐池中暂不更新展示）
  → 后台审核队列 → 通过（进入/更新推荐池）
                  → 驳回（附理由，小程序端展示，会员修改后重新提交）
```

### 5.2 与当前 demo 的主要差异（本期改造重点）

1. **删除演示性模块**：三方商户入驻、交易订单为 demo 演示数据，正式版移除（业务聚焦婚恋主业务，已与甲方确认方向）。
2. **会员视图合并**：将"用户"与"资料"两个分离页面合并为统一会员管理，红娘处理一个会员不再来回切换。
3. **敏感操作全程留痕**：查看联系方式、赠送次数、封禁等操作全部记录审计日志，保障会员数据安全。
4. **门店数据隔离**：红娘仅可见本店会员与工单，总部全局可见。

## 六、权益与规则说明

| 规则 | 内容 |
| --- | --- |
| 注册赠送 | 新会员注册赠送 3 次解锁次数 |
| 消耗规则 | 申请联系 1 人消耗 1 次；被拒/驳回不退还 |

| 审核时效（建议值） | 资料审核 24h 内、撮合工单 48h 内响应（超时后台标红提醒） |

## 七、非功能需求

### 7.1 合规与数据安全

- 用户协议、隐私政策首次使用强制确认；协议文案支持后台更新。
- 会员联系方式、微信 openid 等敏感字段脱敏展示，明文查看需权限 + 审计日志。
- **实名核验不留存身份证件照片**，仅记录核验结论、核验门店与核验人（规避个人信息保护法下的证件留存风险）——此点需甲方确认（见九）。
- 会员可通过反馈渠道申请注销与数据删除。

### 7.2 性能与可用性

- 小程序首屏（每日推荐）加载 ≤ 2s（正常网络）。
- 后台列表页支持 1 万+ 会员规模的分页与筛选。
- 数据每日备份，上传图片与数据库统一持久化目录，支持整体迁移。

### 7.3 技术形态（简述）

微信小程序（原生）+ FastAPI 后端 + PostgreSQL + React 管理后台，自有服务器 Docker 部署，不依赖微信云开发。已具备 HTTPS/多域名生产部署方案。
