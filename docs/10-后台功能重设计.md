# 10 · 后台功能重设计

> 2026-07 起草。前提决策（已与业务确认）：
> 1. **聚焦婚恋主业务**——砍掉"三方入驻 / 交易订单"这条第三方商户线（当前为纯前端 mock）。
> 2. **总部 + 门店两级**权限体系。
> 3. **变现维持"解锁次数"模式**，暂不接支付；订单/对账模块不做，但权益流水要留扩展点。
> 4. 本文档先行，确认后分期实施。

## 1. 现状盘点

| 现有页面 | 数据源 | 处置 |

| items `/items` | 模板遗留（8 行） | **删除** |

核心问题：
- **"用户"和"资料"割裂**：同一个会员的账号状态、资料审核、实名、权益分散在两页，红娘处理一个人要来回跳。
- **审核没有队列心智**：资料审核混在资料列表里，没有"今天还有多少待审"的工作流。
- **门店维度缺失**：Staff/Store 模型已有雏形（`role: staff/store_owner`、`profile.home_store_id`），但后台页面没有按门店隔离的视图。
- **敏感操作无审计**：赠送解锁次数、查看联系方式、封禁，都没有操作日志。

## 2. 角色与权限模型

两级三角色，沿用现有表（User.is_superuser + Staff.role），只做规范化：

| 角色 | 载体 | 说明 |
| --- | --- | --- |
| **总部管理员** | `User.is_superuser=true` | 全部权限，唯一能管员工账号、平台设置 |
| **总部员工** | `Staff.role='hq_staff'`（现 `staff`） | 全局数据可见；可审核、处理工单、处理反馈；不可封禁、不可赠送权益、不可动配置 |
| **门店红娘** | `Staff.role='matchmaker'`（现 `store_owner`，必填 `store_id`） | 仅本店会员（`profile.home_store_id = store_id`）：查看/代录资料、线下实名认证、处理分配到本店的撮合工单 |

### 权限矩阵

| 功能 | 总部管理员 | 总部员工 | 门店红娘 |
| --- | :-: | :-: | :-: |
| 工作台（待办 + 指标） | 全局 | 全局 | 本店 |
| 会员列表 / 详情 | 全局 | 全局 | 本店 |
| 资料审核（audit） | ✅ | ✅ | ❌（线上审核归总部） |
| 实名认证（verify） | ✅ | ❌ | ✅ 本店（线下当面核验） |
| 编辑/代录会员资料 | ✅ | ❌ | ✅ 本店 |
| 查看会员联系方式 | ✅（记审计） | ❌ | ✅ 本店（记审计） |
| 撮合工单处理 | 全部 | 全部 | 分配到本店的 |
| 赠送解锁次数 | ✅ | ❌ | ❌ |
| 封禁 / 解封 | ✅ | ❌ | ❌ |
| 门店 CRUD | ✅ | ❌ | 仅编辑本店信息 |
| 意见反馈处理 | ✅ | ✅ | ❌ |
| 数据统计 | 全局 | 全局 | 本店 |
| 平台设置（资质/公告） | ✅ | ❌ | ❌ |
| 员工账号 / 操作日志 | ✅ | ❌ | ❌ |

## 4. 模块功能明细

### 4.1 工作台（重做 Dashboard）

从"统计数字陈列"改为**待办驱动**，登录第一眼知道今天要干什么：

- **待办卡片**（点击直达对应队列）：待审核资料 N、待处理撮合工单 N、待处理实名申请 N、未处理反馈 N。红娘只看本店口径。
- **核心指标**（次要位置）：今日新注册、今日活跃、累计会员、审核通过率、工单撮合成功率。
- **最近动态流**：最近注册 / 最近提交审核 / 最近工单（各取 5 条）。

### 4.2 会员管理（核心重构：用户 + 资料 合并）

**列表页**：一行一个会员，字段 = 寻缘号、昵称/姓名、性别年龄、所属门店、资料审核状态、实名状态、解锁余额、账号状态、最近活跃。
筛选：审核状态 / 实名状态 / 账号状态 / 门店 / 性别 / 注册时间。搜索：寻缘号、姓名、手机号。

**会员 360° 详情页**（本次重构最大的体验提升），Tab 结构：

| Tab | 内容 |
| --- | --- |
| 资料 | Profile 全字段 + 照片墙 + 完善度；可编辑（管理员/本店红娘） |
| 择偶要求 | Criteria + 父母信息 ParentsInfo；可编辑 |
| 联系方式 | 脱敏展示，点"查看"需二次确认并写审计日志 |
| 权益 | 解锁余额 + BalanceTransaction 流水（赠送/消耗）；管理员可在此赠送 |
| 行为记录 | 收藏了谁/被谁收藏、看过谁/被谁看过、好感、解锁记录 |
| 工单 | 该会员发起/被申请的撮合工单历史 |
| 操作日志 | 后台对该会员的所有操作（审核、封禁、赠送、查看联系方式…） |

**详情页顶部操作条**（按角色显隐）：通过/驳回审核、实名认证/取消、封禁/解封、指定所属门店、赠送次数。

### 4.3 审核中心（新增，队列心智）

两个队列 Tab：

1. **资料审核**：`audit_status=pending` 的队列，卡片式左右对照（新提交内容 + 照片放大），操作 = 通过 / 驳回（必填理由，同步给小程序端展示）。支持键盘快捷键连续审核。
2. **实名认证**：`verified=pending` 的队列（小程序端申请后进入）；总部可远程通过，红娘在门店当面核验后通过，记录 `verified_by_store_id`。

### 4.4 撮合工单（升级现"申请工单"）

补全状态机并可视化：

```
pending（待处理）
  ├─ 红娘联系双方 → accepted（对方同意）→ contacted（已建群/已互推）→ closed
  └─ rejected（对方拒绝）→ closed
```

- 列表按状态分 Tab；每单显示双方会员摘要卡（可点开 360° 详情）。
- **门店分配**：工单默认按发起方 `home_store_id` 归属门店，总部可改派；红娘只见本店工单。
- 处理记录：`handled_by / handled_at / admin_note` 已有，补充展示时间线。
- 逾期提醒：pending 超 48h 标红。

### 4.5 权益管理（新增页面，仅总部管理员）

- **流水列表**：统一的 BalanceTransaction 视图（类型 = 注册赠送 / 后台赠送 / 解锁消耗 / 工单消耗 / 退还），可按会员、类型、时间筛选——替代现在"赠送完无处可查"的状态。
- **赠送操作**：单个赠送（现有）+ 批量赠送（按筛选条件圈人，如"给所有已实名会员送 2 次"）。
- 为将来接支付预留：流水表加 `source` 字段，支付订单未来只是多一种 source。

### 4.6 门店管理（增强）

- 门店 CRUD（现有）+ 门店详情页：本店红娘列表（可在此直接创建红娘账号，代替去"员工"页）、本店会员数、本店工单数、本店实名认证数。
- 红娘视角只读本店信息，可改地址电话照片等展示字段。

### 4.7 意见反馈（增强）

- 状态流：`open → handled`，处理时可填处理备注（新增字段 `admin_note`、`handled_by/at`）。
- 列表可点开关联会员 360° 详情。

### 4.8 数据统计（从 Dashboard 拆出）

- 趋势图：注册、活跃、资料提交、审核通过、工单量（按日/周）。
- 转化漏斗：注册 → 完善资料 → 审核通过 → 发起/收到工单 → 撮合成功。
- 门店对比：各店会员数、实名数、工单成交率（红娘只见本店）。
- 实现：先用现有 `/admin/stats` 扩展成按日聚合接口，不引入额外分析组件。

### 4.9 平台设置（仅总部管理员）

- 资质证明图片（现有 `/qualifications` 迁入）。
- 公告 / 协议文案（用户协议、隐私政策、门店协议、退款说明）——小程序端已有对应页面，后台用 SiteSetting key-value 管理，改文案不用发版。

### 4.10 系统管理（仅总部管理员）

- **员工账号**：现有 Staff CRUD，角色改为 `hq_staff / matchmaker` 下拉，matchmaker 必选门店。
- **操作日志**（新增）：全局审计流水，可按操作人、操作类型、目标会员、时间筛选。

## 5. 数据模型变更

### 新增表

```python
class AuditLog(SQLModel, table=True):
    """后台敏感操作审计."""
    id: uuid.UUID
    actor_type: str          # 'superuser' | 'staff'
    actor_id: uuid.UUID
    action: str              # audit_pass/audit_reject/verify/unverify/block/unblock/
                             # grant_balance/view_contact/update_profile/assign_store/...
    target_user_id: uuid.UUID | None
    detail: dict             # JSON: 变更前后值、理由、金额等
    created_at: datetime
```

```python
class BalanceTransaction(SQLModel, table=True):
    """解锁次数流水（正=入账, 负=消耗）. 未来支付订单只是多一种 source."""
    id: uuid.UUID
    user_id: uuid.UUID
    amount: int              # +2 / -1
    balance_after: int
    source: str              # register_gift / admin_grant / unlock_cost /
                             # contact_request_cost / refund
    ref_id: uuid.UUID | None # 关联 Unlock / ContactRequest / AuditLog id
    note: str | None
    created_at: datetime
```

### 修改

| 表 | 变更 |
| --- | --- |
| `Staff` | `role` 值迁移：`staff → hq_staff`、`store_owner → matchmaker`（alembic data migration） |
| `ContactRequest` | 新增 `store_id: uuid | None`（归属门店，默认取发起方 home_store_id，总部可改派） |
| `Feedback` | 新增 `admin_note`、`handled_by`、`handled_at` |
| `Unlock` / 现有扣减逻辑 | 写入点同步写 `BalanceTransaction`（双写，Unlock 表保留不动） |

不动：User / Profile / Criteria / ParentsInfo / Store / Favorite / View / Affinity / SiteSetting。

## 6. API 变更清单（backend/app/api/routes/admin.py）

### 新增
- `GET /admin/members/{user_id}/full` —— 会员 360°聚合（profile + criteria + parents + 权益 + 行为统计），替代前端拼 N 个请求
- `GET /admin/members/{user_id}/activities` —— 行为记录分页（favorites/views/affinity/unlocks）
- `GET /admin/members/{user_id}/transactions` / `GET /admin/transactions` —— 权益流水
- `POST /admin/grants/batch` —— 批量赠送
- `GET /admin/verify-queue` —— 实名认证队列
- `POST /admin/contact-requests/{id}/assign` —— 工单改派门店
- `GET /admin/audit-logs` —— 操作日志
- `GET /admin/stats/daily?from=&to=` —— 按日聚合统计
- `PUT /admin/feedback/{id}` —— 反馈处理（备注+状态）
- `GET/PUT /admin/site-settings/{key}` —— 通用配置读写（公告/协议）

### 修改
- `GET /admin/profiles`、`/admin/users` → 合并为 `GET /admin/members`（一个列表接口带全部筛选）；旧接口过渡期保留
- 所有敏感操作端点（audit/verify/block/grant/联系方式读取）内部写 `AuditLog`
- 红娘鉴权：现有 `_check_store_scope` 扩展到工单、统计接口

### 删除
- 无需删除后端（merchants/orders 本来就没有后端）；前端删 `merchants.tsx`、`orders.tsx`、`items.tsx` 及路由/侧边栏项

## 8. 待确认事项
